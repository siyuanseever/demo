## iOS App 原型方向

当前 Web UI 已经可以验证后端智能机制，但心理陪伴类产品更适合在手机上形成触摸、语音、动画和随身陪伴体验。下一阶段不直接做完整上架产品，而是先做一个复用现有后端的 iOS 体验原型。

### 阶段 0：开发环境准备

- 安装 Xcode，用于 SwiftUI 开发、iOS 模拟器、本机调试和真机安装测试。
- 准备一台 iPhone 作为真机测试设备。
- 前期不急着处理 App Store 上架、付费、账号系统和复杂合规，只先验证体验。
- 后续如果需要上架，再准备 Apple Developer Program、隐私政策、应用图标、截图、审核说明和数据安全说明。

### 阶段 1：最小 iOS 聊天壳

目标：让 iOS 只负责体验外壳，继续调用当前 Python 后端。

- 新建 SwiftUI App。
- 实现基础聊天流：
  - 用户输入。
  - 调用本地/局域网后端接口。
  - 展示四段式回复：感受共鸣、需求点明、主回复、锚点一句话。
- 复用现有六个小动物头像和基础配色。
- 暂时不做账号、支付、云端同步和 App Store。

### 阶段 2：触摸与小动物陪伴感

目标：验证“手机上的小动物是否更有陪伴感”。

- 每只小动物可以被触摸。
- 触摸后给出轻量反馈：
  - 眨眼。
  - 点头。
  - 歪头。
  - 贴近。
  - 发光。
  - 轻轻跳一下。
- 不先做复杂动画系统，先用 SwiftUI 简单动画或少量序列帧验证。
- 后续再考虑透明通道素材、Live2D、Lottie、Rive 或 SpriteKit。

### 阶段 3：语音对话

目标：让用户可以用说话的方式与小动物交流。

- 语音转文字 STT：
  - 用户按住或点击麦克风说话。
  - 转成文本后进入现有对话流程。
- 文本转语音 TTS：
  - 每只小动物拥有独立声线。
  - 声线需要匹配角色性格、形象和回复类型。
- 第一版可以先使用系统语音或单一 TTS 服务验证流程。
- 第二版再为六个角色定制不同声线：
  - 绵绵羊：柔软、慢、安定。
  - 石石龟：低稳、可靠、温和。
  - 墨墨鸦：冷静、聪明、带一点轻微幽默。
  - 忧忧兔：轻、软、怯生但真诚。
  - 闪闪蝶：明亮、轻快、有希望感。
  - 敢敢虎：坚定、暖、保护感强。

### 阶段 4：即时反馈层

当前流程是：

1. 本轮规划器：理解用户状态、需求、风险、角色、检索关键词。
2. 内容生成器：生成四段式回复。

移动端语音体验中，等待时间会更明显，因此可以在两轮之间加入一个即时反馈层：

1. 用户说完话后，先给一个极短的即时回应。
2. 同时后台继续做本轮规划、记忆检索、知识卡检索和主回复生成。
3. 主回复准备好后，再补充更完整的多角色回应。

即时反馈的形式可以是：

- 小动物动作：抬头、贴近、点头、歪头、眨眼。
- 一句很短的话：例如“我听见了”“这听起来真的很累”“先让我陪你停一下”。
- 一个轻量声音反馈：小动物轻声应答，不急着分析。

注意：即时反馈不能抢先下结论，也不能在安全风险不明时给过度安抚。它更像“我在听”，不是正式分析。

### 阶段 5：疗愈互动内容

当本轮规划器判断用户不只需要文字回复时，可以引导进入更具体的互动内容。

- 情绪天气卡：把当前感受可视化成天气。
- 呼吸练习：由某只小动物带领做 30 秒到 2 分钟呼吸。
- 身体扫描：适合冻结、焦虑、脑子很累时。
- 小纸条：生成一句可保存的小动物留言。
- 冥想片段：短音频或图文引导。
- 轻量小游戏：例如整理情绪碎片、给小动物盖毯子、把担心放进盒子。
- 心理知识卡：当用户需要理解机制时展示，不每轮强推。

### 阶段 6：产品化与上架

只有当 iOS 原型证明体验成立后，再进入产品化。

- 账号与数据同步。
- 云端部署和 HTTPS。
- 隐私政策、用户协议、免责声明。
- 危机风险处理和安全边界。
- App Store 图标、截图、介绍文案。
- Apple Developer Program 和 TestFlight 内测。
- 上架审核材料。

## LLM 多轮判断循环方向

后续一次用户输入可以不再只是“直接生成回复”，而是拆成多轮模型判断：

1. `InputInterpreter`：理解用户说了什么。
2. `EmotionAnalyzer`：识别情绪、强度、身体感、关系触发。
3. `NeedAnalyzer`：判断用户此刻更需要共情、稳定、洞察、边界、行动还是轻盈感。
4. `RoleRouter`：选择一只或多只小动物。
5. `MemorySelector`：选择此刻适合使用的长期记忆。
6. `KnowledgeSelector`：判断是否需要心理学知识卡或练习卡。
7. `StateUpdater`：更新每只小动物的心情、状态和内心想法。
8. `ResponsePlanner`：控制总字数、结构和语气。
9. `Responder`：生成最终回复。
10. `TraceWriter`：把本轮判断过程保存下来，供开发者和用户查看。

目标不是每次都调用很多轮模型，而是建立一个可逐步打开的框架：简单场景走轻量路径，复杂场景走更完整的 agent 流程。

## 群聊自动 v1：大模型选角

当前群聊自动只是关键词检测，成本低、速度快，但理解能力有限。下一版可以改成 LLM 路由器。

目标流程：

1. 用户输入文本。
2. 调用大模型做情绪和意图判断。
3. 大模型输出结构化 JSON：
   - `dominant_emotion`
   - `emotion_intensity`
   - `support_need`
   - `recommended_character_id`
   - `reason`
4. 后端根据推荐角色生成最终回复。
5. 前端展示“本轮由谁回应”和简短理由。

注意事项：

- 这会多一次模型调用，成本和延迟都会增加。
- 需要稳定的 JSON 输出和失败兜底。
- 如果路由器失败，应回退到当前关键词规则。
- 选角 prompt 需要单独版本管理和测试用例。

## 群聊自动 v2：多阶段 Agent 编排

更高级的版本可以参考 Claude Code / OpenClaw 的 agent 思路，把一次回复拆成多个步骤。

候选流程：

1. `EmotionAnalyzer`：识别情绪、强度、身体状态、关系触发。
2. `RoleRouter`：判断适合哪只小动物，或是否需要多只共同回应。
3. `KnowledgeRetriever`：检索心理学知识卡、内容卡、练习卡。
4. `ResponsePlanner`：规划回复结构和总字数。
5. `AnimalResponder`：按角色生成回复。
6. `SafetyGuard`：检查风险、边界和依赖性语言。
7. `MemoryWriter`：决定是否写入长期记忆。

设计原则：

- 不追求“每次都多 agent”，只在必要时多步。
- 控制 token 和延迟，避免一次普通对话变得太重。
- 每一步都要有日志和可视化反馈，让开发者知道发生了什么。
- 多角色回复要控制总字数，不能简单把多个完整回复拼起来。

## 多角色群聊 v3：多只小动物共同回应

未来可以让 1-3 只小动物一起出现，但必须控制信息密度。

可能模式：

- 主回复：一只小动物负责完整回应。
- 共情补充：一只小动物只说一句情绪陪伴。
- 行动补充：一只小动物只给一个小下一步。
- 表情/动作：一只小动物只提供短动作，例如“忧忧兔轻轻点头”。

约束：

- 总字数最多比现在单角色回复多一点，不能翻倍。
- 默认 1 只，必要时 2-3 只。
- 不要让角色互相抢话。
- 不要让用户感觉被很多声音包围。

## 当前阶段：Mac 应用体验整合与稳定化

目标是把现有 Mac 应用从“已有页面和首轮功能”推进到“可稳定使用、数据完整、心流与夜谈形成闭环”。优先级如下：

1. **P0 性能与鲁棒性**：先解决卡死、点击迟缓和主线程阻塞。
2. **P0 数据与 UI 完整性**：已有数据库内容必须能够在 Mac UI 中正确、分层地访问。
3. **P1 心流 ↔ 夜谈互融**：两个核心界面双向展示相关内容，并能点击回到来源或详情。
4. **P2 体验收敛**：保持温暖、治愈、流动、简洁，不增加无关内容和精神负担。

Web/Python 当前只做必要兼容、缺陷修复和数据契约支持，不作为主动扩展主线。

---

## 阶段范围

### 做

- 验证现有 Mac target 的构建、启动和关键交互路径。
- 建立性能基线，修复可复现卡死和慢响应。
- 建立数据库/API 到 Swift UI 的字段映射矩阵。
- 补齐长期记忆类别、记忆详情、会后总结和三篇关联日记的展示。
- 完成心流与夜谈的双向摘要、导航、更新和空状态。

### 不做

- 无关 Web 功能、语音、账号、支付、云同步和新小游戏。
- 与三条主线无关的整体重构或视觉重做。
- 未经用户确认的心理安全策略和数据契约变更。
- 以随机弹窗、密集卡片或持续提醒制造“互动感”。

## 验收标准

| 领域 | 完成条件 | 必需证据 |
|---|---|---|
| Mac 构建 | 目标可成功构建并启动 | Xcode/xcodebuild 命令、目标和退出码 |
| 稳定性 | 关键路径无可复现卡死；无超过 1 秒主线程停顿 | 场景、数据规模、优化前后耗时或 trace |
| 点击响应 | 关键点击立即出现视觉反馈，异步操作不阻塞界面 | 人工操作记录或 UI 自动化证据 |
| 心流 → 夜谈 | 夜谈按规则展示最多一条相关心流摘要，可进入详情 | 触发规则、空状态、导航和更新场景 |
| 夜谈 → 心流 | 心流可查看近期夜谈形成的状态/总结/模式并进入来源 | 真实数据映射和导航场景 |
| 长期记忆 | 数据库现有类别全部可访问，核心字段不被静默丢失 | 字段矩阵 + 代表性数据截图/记录 |
| 会后总结 | 摘要及其三篇关联日记可查看，缺失数据有降级状态 | 关系映射 + 完整/缺失两类场景 |
| 设计原则 | 默认界面信息克制，详情按需展开，无无关内容 | 人工体验审查 |
| Python 兼容 | 后端相关改动不破坏现有 Gate | `compileall` + evaluation runner |

无法取得某项必需证据时，该项必须标为 `pending_manual_validation` 或 `blocked`，不能标为完成。

## 自动化执行约束

- PM 每日最多下发一个可执行 Mac 任务；任务必须包含工作流、目标页面、数据来源、非目标和验收证据。
- Executor 不写测试、不修改规划；没有 Xcode/Mac 验证能力时不得执行依赖该能力才能安全完成的任务。
- Checker 独立复验功能、数据映射和性能证据；Python Gate 不能替代 Mac 验收。
- Fixer 只修复 Checker 已稳定复现且契约明确的产品缺陷。
- 安全策略、数据迁移、删除用户数据和大范围架构调整必须由用户确认。

## 主要风险

| 风险 | 应对 |
|---|---|
| Swift 工程提交后未在目标 Mac 编译 | 把构建证据设为强制门控；无环境即阻塞 |
| 为“互动”加入过多提示 | 默认一条摘要、明确冷却、按需展开 |
| 数据字段补齐导致页面拥挤 | 先做字段矩阵，再设计摘要/详情层级 |
| 性能优化凭感觉完成 | 基线、复现、同场景复测三者缺一不可 |
| 自动化任务扩大范围 | PM 每日一个小任务，明确非目标和路径边界 |

## 当前阶段：回复价值验证与 Mac 产品化

当前实现仍是 **iOS App 的 Mac Catalyst 迁移版**，不是原生 macOS App。近期已处理消息列表内存增长、双阶段回复卡死，并完成 Conversation Trail v1。但当前最重要的产品信号是：用户仍然不愿意以真实用户身份持续对话，因为回复经常只有温柔复述，缺少信息增量。接下来的核心不是继续堆 UI，而是证明“小兔子的下一句话确实能帮助用户”。

优先级：

1. **硬门槛：稳定性回归线**：连续发送、长列表滚动和内存占用不得因新能力回退。
2. **唯一产品 P0：`RESPONSE-VALUE-001`**：回复至少提供被理解、认知增量、行动价值、内容价值中的一种，而不是只做语言陪伴。
3. **已完成 v1：Conversation Trail / 对话足迹**：停止继续扩展普通 UI，先验证现有轻量轨迹的稳定性和真实使用价值。
4. **当前用户指定主线：语音输出（TTS）**：先完成免费系统女性声线 V0；回复价值人工评分暂缓，后续由 Agent 先做自动初评。
5. **P1-P2：原生 macOS**：N0 架构盘点可以并行；N1 代码迁移在回复与发送共同逻辑稳定后推进。

Web/Python 只做 Mac 所需的数据契约、智能能力和缺陷修复，不主动扩展无关 Web 功能。

## 当前四个产品重点

### 1. `RESPONSE-VALUE-001`（当前唯一产品 P0）

产品目标不是“更长、更专业、更像心理咨询师”，而是让每一轮回复产生可感知的新价值。小兔子应逐步成为一个能够利用长期个人历史，持续形成、验证和修正“关于这个人的理解”，并在对话中产生真实认知增量的心理陪伴 Agent。

每轮回复至少命中以下一种价值：

- **被理解的价值**：准确指出真正卡住的地方，而不是换一种说法复述。
- **认知增量**：发现模式、矛盾、因果、隐藏前提或跨时间联系。
- **行动价值**：给出适合这个用户、这个时刻的具体下一步，避免万能建议。
- **内容价值**：引入真正相关的心理概念、书、电影、音乐或知识卡，并解释它为何相关。

明确避免：空泛安慰、未经证据的心理解释、模板化“散步/呼吸/写日记”、为了显得专业而堆术语、仅仅延长 deep reply。

#### R0：建立真实回复价值评估集

- 从既有真实会话中选择 20–30 条代表性输入，覆盖求职、关系失落、梦境、低质量社交、学习与设计冲突、家庭金钱、身体疲惫和孤独。
- 原始私人文本只保存在本地私密评估文件或数据库中，不提交 Git，不写入自动化日志。
- 当前版本完整跑一遍，建立改造前基线；不要用为了测试而编造的假输入。
- 每项采用 0–2 分并保留人工备注：复述比例、信息增量、隐藏结构、历史记忆使用、建议个性化、AI 套话、是否让人想继续说。
- 同时保留安全否决项：编造记忆、诊断化、强行积极、危机场景处理错误。

现有实现可复用 `app/evaluation/dataset_from_db.py`、`manual_eval.py` 和 rubric，但需要先做以下改造：

- 新建独立的回复价值数据集入口，不改变现有 intent benchmark 的语义。
- 按用户消息和 `reply_group_id` 聚合一整个 turn；quick、deep、clarify 或 interaction 都必须进入同一案例，不能只取第一条 assistant 回复。
- 输出默认放在已被 `.gitignore` 排除的 `data/` 或 `eval_reports/`，报告只使用本地 case ID，不把私人原文复制到日志。
- 先自动生成候选案例，再由用户确认最终 20–30 条；不要完全随机采样，需覆盖不同主题、回复路径和是否命中记忆。
- 第一版以人工评分为准；自动评分只能用于提示“可能存在套话/复述/无证据记忆”，不能替代用户对真实价值的判断。

#### R1：Plan 升级为 Response Strategy Agent

Plan 不再只输出 `intent/emotion/next_action`，还要形成可审计的回复策略：

```text
surface_emotion
core_tension
hidden_conflict
cross_memory_patterns[]
response_value_target       # understanding | insight | action | content
candidate_insight
supporting_memory_ids[]
avoid[]
confidence
next_action                 # deep | quick_only | clarify | interaction
```

- `quick` 继续与 strategy 并行，负责即时接住用户，不等待复杂分析。
- strategy 完成后决定是否需要 deep 或其他 action，并把“本轮要创造什么价值”传给生成器。
- `candidate_insight` 是待验证假设，不是真理；deep 必须用用户原话和记忆证据校验，证据不足时改为澄清。

#### R2：让长期记忆参与推理

- Memory 不只作为 prompt 附件，而要支持跨时间模式、矛盾证据和变化趋势。
- 检索结果必须携带 memory ID、时间、置信度和来源 session，便于策略层引用和审计。
- 对相互矛盾的记忆保留并列证据，不强行合并成单一结论。
- 评估“第 299 次对话是否真的比第一次更懂用户”，而不是只统计记忆数量。

#### R3：验收与迭代

- 先人工评估当前基线，再修改 strategy prompt；每次只改变一个主要变量。
- 同一批案例做前后盲评，并记录回复价值提升来自哪一层：策略、记忆、知识还是生成。
- 达到可接受基线前，不把 TTS、原生迁移或更多视觉功能当成主开发线。

### 2. Conversation Trail / 对话足迹（v1 已完成）

- 默认只显示窄轨迹，不侵占聊天阅读空间。
- 每条轨迹对应一次用户提问；同一轮的 `quick → deep` 只占一个 marker。
- 悬停 marker 时横条变长；预览只保留用户问题和本轮回答摘要，不渲染第二份完整消息正文。
- 点击条目使用原消息 ID 定位，不复制第二份完整消息列表。
- 预览文本必须截断并限制渲染数量，继续遵守 `b4ff586` 建立的列表内存边界。
- 用户手动滚动时，后续版本可高亮当前轮次；知识卡、长期记忆、情绪强度等 marker 装饰暂缓。
- v1 完成后冻结功能扩展，先完成 `RESPONSE-VALUE-001`。

### 3. 语音输出（V0 已实现，等待试听确认）

先建立统一 `SpeechService`，UI 只依赖“播放、暂停、停止、自动朗读”接口，再替换具体声音来源。

| 路径 | 成本与复杂度 | 声音效果 | 当前判断 |
|---|---|---|---|
| 系统 `AVSpeechSynthesizer` | 免费、接入最快、无需部署 | 稳定但“幼幼兔”个性有限 | 作为 V0 播放链路原型 |
| 本地开源 TTS | 无按次费用，但模型体积、签名和性能成本高 | 可定制，需实际筛选中文模型 | V1 做独立实验，不先打包进主 App |
| 云端 TTS API | 按量付费、需要网络和密钥管理 | 通常最自然，容易选可爱音色 | 与 V0 做盲听和成本对比后决定 |

V0 已完成每条兔子回复旁的播放/停止按钮、设置页试听和“自动朗读新回复”开关；默认不自动朗读。当前优先使用本机 `Tingting/婷婷` 普通话女性声线，未安装时自动选择其他中文女性声线。下一步需要人工试听声音、语速、音高和长回复打扰感，再决定是否进入本地开源模型实验。

### 4. 原生 macOS（N0 并行准备）

- N0 先列出 SwiftUI 可共享层、UIKit/Catalyst 专属层、数据库和网络边界。
- N1 建立原生 macOS target，只接启动、导航、设置、日志和空夜谈页。
- N2 迁移夜谈纵切时复用同一 quick/plan 调度、对话轨迹和关联 ID。
- Catalyst 在原生版本通过连续发送、历史定位和内存回归前继续保留，不做一次性重写。

## 已确认的技术边界

- 当前使用 Catalyst 作为稳定化基线；长期迁移原生 macOS，但不以未经取证的重写代替卡死修复。
- Python 后端 `data/app.db` 是既有数据权威源。
- Mac App 使用沙盒 Documents 中的独立 `app.db` 作为缓存和离线降级。
- App 不直接持续读写仓库数据库文件；通过本机 API 自动增量刷新。
- 正常使用不依赖手动同步按钮；按钮保留为故障恢复入口。
- 完全脱离 Python 后端是未来独立架构项目，需要单独决策，当前不同时维护两套权威业务逻辑。

## 验收标准

| 领域 | 完成条件 | 必需证据 |
|---|---|---|
| 自动化 | 运行时使用 v3；每个 schedule slot 仅一个 run；每个 task 仅执行一次 | state/index、slot_id、claim 记录 |
| Mac 构建 | Catalyst 构建并启动，关键页面可进入 | xcodebuild 命令、退出码、启动存活证据 |
| 稳定性 | 关键路径无可复现卡死；无超过 1 秒主线程停顿 | 数据规模、复现步骤、trace、前后对比 |
| 内存安全 | 无持续单调增长；重复操作后回落；自动场景不触发 2GB 保护 | resident/footprint 曲线、Allocations、memgraph |
| 发送链路 | 点击发送后 UI 保持响应，阶段可追踪，后端在线时出现同一 correlation ID | App 阶段日志、后端接收日志、heartbeat |
| 自动同步 | 启动、回前台、写入后、后端恢复触发去重刷新 | 触发矩阵、网络/日志、缓存与更新场景 |
| 离线降级 | 后端不可用时立即显示缓存，不阻塞，不丢失本地可见内容 | 离线和恢复场景 |
| 长期记忆 | 一级类别、二级类别、叶节点详情均可点击到达 | 字段矩阵和导航记录 |
| 最近更新 | 新增/更新记忆和日记在刷新后可见 | 后端写入 → 自动刷新 → UI 证据 |
| 会后总结 | 摘要及三篇关联日记可进入；缺失数据有降级状态 | 完整/缺失两类场景 |
| 心流单卡片 | 主界面/夜谈每次最多一张相关卡，可进入来源 | 选择、冷却、导航、空状态 |
| 轻量行动 | 每卡最多一个可选动作；点击即时正反馈并记录；忽略无后果 | 点击/不点击两类场景 |
| 回复价值 | 真实案例产生可解释的信息增量，并正确使用历史证据 | 本地案例集、人工盲评、策略 trace、前后对比 |

无法取得必需证据时必须标记 `pending_manual_validation` 或 `blocked`，不得标记完成。

## 自动化执行约束

- 当前 P0 模式：PM 每天 1 次；Executor 每天 1 次主执行 + 1 次仅重试 slot；Checker/Fixer 每 6 小时错峰运行。
- 稳定 7 天后：Checker 降为每 12 小时，Fixer 降为每天 1 次。
- PM 每个日程槽最多生成一个可执行任务；其余只写 backlog 建议。
- PM 在 main 只读运行，只写 handoff，不修改或提交仓库文件；其他三个 Agent 固定使用 automation worktree。
- Executor 必须先原子 claim 任务；旧协议、多任务、重复任务、错误分支一律拒绝执行。
- Checker 每 6 小时复验新产品提交和证据，不重复消费同一 Executor/Fixer run。
- Fixer 只处理 Checker 已复现的产品缺陷，不承担 Roadmap 功能开发。
- Git 同步是确定性前置步骤，不得由 PM 作为产品任务派发。
- 平台方向、数据权威源、迁移、安全策略和删除数据必须由用户或本文件明确。
- worktree/branch 生命周期只归用户/bootstrap；四个 Agent 发现缺失时必须停止。

## 主要风险

| 风险 | 应对 |
|---|---|
| App 与 Python 同时写同一数据库 | 使用 API + 独立沙盒缓存，不共享活动数据库文件 |
| 自动刷新引发更多卡顿 | 缓存先显示、增量拉取、去重、取消、后台事务 |
| 同一自动化任务重复执行 | schedule slot + task claim + completed key 三层幂等 |
| PM 从代码片段推断产品方向 | 方向不在 Roadmap/plan 时只能 `needs_human` |
| 点击互动变成任务负担 | 每卡一个可选动作；不点击无提醒、无惩罚 |

# iOS 产品方向记录

这个文档用于记录当前 iOS App 的产品方向和工程约束，避免后续实现逐渐偏离目标体验。

## 开发目标系统

- 后续设计和验证优先使用 iOS 26.5。
- 当前优先模拟器：iPhone 17 Pro，iOS 26.5。
- 不再用 iOS 18.x 的视觉表现判断最终样式，因为新旧系统的材质、控件和整体观感差异很大。

## 核心体验：宁静的沉浸感

这个产品的目标不是“又一个聊天 App”。主体验应该是宁静的沉浸感：

- 一个安静的深夜世界。
- 六只小动物围在篝火旁。
- 有月光、云、星星、火光、草地，后续可以有天气。
- 聊天文本不应该占据第一视觉。
- 历史消息可以收进一个沉浸在场景里的小物件，比如信箱、笔记本、灯、树洞。
- 主界面应该像进入一个温柔的游戏世界，而不是打开一个普通工具界面。

当前代码生成的篝火、月亮和动物布局只能作为早期脚手架。后续视觉需要逐步替换成更高质量、符合固定画风的 raster 资产。

## 界面洁净原则

宁静的沉浸感需要同时保持足够干净。主界面优先出现有具象含义的元素，比如夜空、树木、月亮、云、草地、篝火、小动物、萤火虫、信箱、灯笼、笔记本、树洞等；尽量减少没有场景意义的普通控件，比如标题栏、孤立按钮、输入框、说明文字和工具面板。

功能可以先做完整，但后续要逐步收进场景物件里。用户看到的第一层应该是一个安静世界，不是功能按钮集合。文字和工具可以存在，但应该隐藏在信箱、笔记本、灯、树洞这类可触摸物件之后，只有用户主动触摸时才浮现。

临时代码场景也要遵守基本构图：天空、远景、火光、小动物、前景物件应有清晰层级，不要把可点击物件随意堆在一起。即使未来会换成 Image Gen 资产，当前脚手架也应该先保持安静、留白和不重叠。

Image Gen 首页背景可以分阶段推进：第一阶段使用一张无文字竖屏氛围图作为背景，并在 SwiftUI 里叠加透明触摸热区；第二阶段逐步把小动物、篝火、灯笼、信箱、笔记本等拆成独立透明资产；第三阶段再做天气、动作、触觉和动画。不要一开始就要求整张图内所有物体天然可交互。

可交互边界也分阶段推进：

- 第一阶段：背景只负责环境，小动物和关键物件使用独立 alpha 贴图叠加，触摸区域贴近贴图外框和主要轮廓。
- 第二阶段：为不同物件使用更合适的命中形状，比如圆形、胶囊、组合形状，减少透明区域误触。
- 第三阶段：如果画风和资源稳定，再实现像素级 alpha 命中。做法是在 UIKit 层读取图片 alpha，透明像素不触发点击；这更精确，但维护成本和性能成本都更高。

当前不建议直接从第三阶段开始。先让角色和物件成为独立图层，才能自然支持缩放、发光、轻微呼吸、点按反馈和后续动画。

## 视觉资产和 Image Gen 方向

`小动物夜谈会固定画风.md` 是角色和画风一致性的主要依据。

后续 Image Gen 优先生成这些项目内资产：

- 六只小动物的全身或半身透明 PNG/WebP。
- 有清晰边界的篝火资产。
- 月亮、路灯、信箱、草地、云、雨、星星等场景物件。
- 夜晚天气背景的不同版本。
- 适合作为触觉按钮的前景物件，要求轮廓清晰、带 alpha 通道。

透明图生成策略：

- 默认优先用 Image Gen 生成纯色 chroma-key 背景，再本地去背生成 alpha PNG/WebP。
- 只有当 chroma-key 去背质量不够时，再考虑真正原生透明背景的 fallback 路线。

可触摸物件方向：

- 点小动物：选择角色、触发语音或小反应。
- 点篝火：发送语音、进入安定练习或触发夜谈。
- 点月亮、天气、灯、信箱：作为场景内控制入口。
- 可见物体可以小，但点击热区需要足够大；萤火虫这类物件可能更适合作氛围，不适合做主要按钮。

## 天气和氛围系统

天气属于后续较大的 UI 系统：

- 夜晚可以是晴天、阴天、雨天、星空。
- 月亮、云、雨、星星、火光可以跟真实天气或用户状态发生联系。
- 天气只作为氛围，不应变成信息噪音。
- 天气变化要慢、平静、低刺激。

## 当前 Bug 和 UX 优先级

近期优先级：

- 输入键盘收起和输入体验。
- 群聊模式按钮和状态提示。
- 结束 session、总结 session、提取记忆、开始新 session。
- 记忆页可点击并进入详情。
- 禁用意外 dark mode，或者做一套明确的固定视觉主题。
- 继续把聊天文本从主视觉中收纳进场景物件或抽屉，让第一屏保持沉浸感。

## 隐私和上线模型

优先考虑 local-first 和无自有服务器：

- 免费基础版。
- 用户数据保存在用户自己的 iPhone。
- 普通使用不依赖自有后端服务器。
- 用户自己填写 LLM API key，例如 DeepSeek API key。
- 可选 iCloud 同步，用用户自己的 iCloud 做备份和跨设备同步。
- 尽量让 Apple/iCloud 承担数据保存和同步复杂度。

这个路线可以降低服务器成本、隐私风险和运维负担。

## API Key 门槛问题

最大风险是：让用户自己填写 API key 会挡住很多人。

- 很多用户不知道 API key 是什么。
- 很多用户不愿意注册、充值、维护第三方模型账户。
- 免费基础版可能需要一个离线/demo/local guidance 模式。
- 后续如果做 app-managed 模型额度或订阅，会重新引入后端、支付、滥用防护和成本控制问题。

当前可探索路线：

- 第一阶段用 BYO API key 作为技术上线路径。
- 提供本地 demo 模式或离线陪伴模式，让没有 API key 的用户也能体验基本界面。
- 提供 DeepSeek API key 的引导式设置。
- 未来收费点可以是小动物语音、更丰富的游戏、早晚安冥想、高质量场景包、或者 app-managed 模型服务。

## 提交纪律

后续实现按小批次提交：

- 每个 commit 对应一个清晰变更集合。
- 不把无关 UI、后端、资产、产品文档混在同一个 commit。
- 每轮实现和验证后，先 commit，再继续下一批。
